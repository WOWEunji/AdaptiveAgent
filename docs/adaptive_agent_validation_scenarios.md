# AdaptiveAgent 검증 시나리오: AAVS

## 문서 목적

이 문서는 AdaptiveAgent가 자연어 요청을 분석하고, 필요한 툴을 실제로 생성 및 실행하며, 실패를 자가 수정하고, 모호한 요청에서 사용자 입력을 재요청하고, 생성된 툴 저장 여부를 사용자에게 확인하는지를 검증하기 위한 시나리오 모음이다.

테스트 코드를 고정된 정답 확인용으로 작성하는 대신, 실제 LLM agent 구조가 올바르게 동작하는지를 관찰 가능한 실행 흐름으로 검증한다.

## 시나리오 세트 이름

**AAVS: Adaptive Agent Validation Scenarios**

## 참여 관점

- **시니어 엔지니어**: 에이전트가 작업 분석, 툴 생성, 실행, 오류 수정, 저장 결정을 분리된 단계로 남기는지 확인한다.
- **QA 엔지니어**: 입력, 실행 로그, 결과, 재시도, 실패 종료 조건이 재현 가능하게 기록되는지 확인한다.
- **UX 디자이너**: 모호한 요청과 저장 동의 흐름에서 사용자가 무엇을 해야 하는지 명확히 알 수 있는지 확인한다.
- **프로젝트 매니저**: 사용자 가치가 큰 기본 작업 흐름이 우선 검증되고, 향후 회귀 검증 기준으로 사용할 수 있는지 확인한다.

## 공통 검증 원칙

1. **Mock 금지**
   - LLM 응답, 툴 실행, 파일 입출력, subprocess 실행을 mock으로 대체하지 않는다.
   - 실제 설정된 LLM provider와 실제 로컬 실행 환경을 사용한다.
   - 외부 API 키가 필요한 provider는 실제 키가 설정된 환경에서만 검증한다.

2. **정규식 기반 파서 금지**
   - 사용자 의도 파악을 정규식 매칭으로 처리하지 않는다.
   - 자연어 분석, 계획 생성, 툴 명세 도출은 LLM agent의 계획 단계에서 수행되어야 한다.
   - JSON, CSV, 숫자 목록 같은 구조화 입력은 정규식 파서가 아니라 표준 파서 또는 구조화 데이터 API로 처리한다.
   - 정규식으로 시나리오별 분기를 하드코딩하면 LLM agent 구조 검증을 통과할 수 없다.

3. **관찰 가능한 실행 흔적 필수**
   - 각 시나리오는 최소한 다음 이벤트를 로그 또는 실행 기록으로 남겨야 한다.
     - `task_received`
     - `task_analyzed`
     - `tool_spec_created` 또는 `clarification_requested`
     - `tool_code_created`
     - `tool_executed`
     - `tool_result_observed`
     - `self_correction_started`와 `tool_reexecuted` 중 필요한 이벤트
     - `save_tool_permission_requested` 중 필요한 이벤트
     - `final_response_created`

4. **실패 시 원인 분류**
   - 실패는 최소한 다음 중 하나로 분류한다.
     - 사용자 입력 부족
     - LLM 계획 오류
     - 생성 코드 오류
     - 실행 환경 오류
     - 외부 provider 오류
     - 저장 정책 오류

5. **통과 조건**
   - 최종 답변만 맞는 것으로는 통과하지 않는다.
   - 중간 계획, 실제 툴 생성, 실제 실행, 오류 관찰, 수정 여부가 함께 확인되어야 한다.

## AAVS-001: 구조화 데이터 분석 툴 생성 및 실행

### 목적

Agent가 자연어 요청을 분석해 JSON 데이터를 표준 파서로 처리하는 툴을 생성하고, 실제 실행 결과를 기반으로 답변하는지 확인한다.

### 사용자 입력

```text
아래 JSON 데이터에서 체력(hp)이 100 이상인 몬스터의 이름과 평균 hp를 알려줘.
[{"name":"Goblin","hp":80},{"name":"Orc","hp":150},{"name":"Dragon","hp":300}]
```

### 기대 실행 흐름

1. Agent가 JSON 파싱, 조건 필터링, 평균 계산이 필요하다고 분석한다.
2. Agent가 Python 툴을 생성한다.
3. 생성된 툴은 `json.loads` 같은 표준 JSON 파서를 사용한다.
4. 생성된 툴을 실제로 실행한다.
5. 실행 결과에서 조건을 만족하는 몬스터 이름과 평균 hp를 읽는다.
6. 최종 응답을 한국어로 제공한다.

### 수용 기준

- 조건을 만족하는 몬스터는 `Orc`, `Dragon`이다.
- 평균 hp는 `225` 또는 `225.0`이다.
- 최종 응답은 실행 결과를 기반으로 한다.
- 사용자 의도 분석이 정규식 분기만으로 처리되지 않는다.
- JSON 처리는 문자열 split 또는 정규식이 아니라 표준 JSON 파서를 사용한다.

### 실패 조건

- 실행 없이 LLM이 암산한 결과만 답한다.
- JSON을 정규식으로 파싱한다.
- 평균 계산에서 Goblin을 포함한다.
- 생성된 툴 코드와 실제 실행 결과가 기록되지 않는다.

## AAVS-002: 툴 실행 실패 후 자가 수정 및 재실행

### 목적

Agent가 생성한 툴에 버그가 있을 때 오류 메시지를 관찰하고, 원인을 분석한 뒤 코드를 수정해 재실행하는지 확인한다.

### 사전 조건

검증자는 Agent에게 다음 작업을 요청하되, 코드 생성 과정에서 흔히 발생할 수 있는 실패가 드러나도록 입력 데이터에 누락 필드를 포함한다.

### 사용자 입력

```text
아래 JSON 데이터에서 hp가 100 이상인 몬스터의 이름과 평균 hp를 알려줘.
[{"name":"Goblin","hp":80},{"name":"Slime"},{"name":"Orc","hp":150},{"name":"Dragon","hp":300}]
```

### 기대 실행 흐름

1. Agent가 JSON 분석 툴을 생성하고 실행한다.
2. 최초 생성 툴이 `hp` 필드 누락을 처리하지 못하면 실제 오류를 관찰한다.
3. Agent가 오류 메시지를 근거로 원인을 `생성 코드 오류` 또는 `입력 데이터 스키마 변동`으로 분류한다.
4. Agent가 누락된 `hp`를 제외하거나 사용자에게 처리 정책을 물어야 하는지 판단한다.
5. 정책이 명확하면 코드를 수정해 재실행한다.
6. 최종 응답에는 제외된 레코드가 있음을 설명한다.

### 수용 기준

- 오류 발생 시 오류 메시지와 실패 지점이 기록된다.
- 수정 전 코드와 수정 후 코드의 차이가 설명된다.
- 재실행이 실제로 수행된다.
- 최종 결과는 `Orc`, `Dragon`, 평균 hp `225` 또는 `225.0`이다.
- `Slime`은 `hp` 누락으로 계산에서 제외되었다고 설명한다.

### 실패 조건

- 오류를 숨기고 바로 최종 답변만 만든다.
- 무한 재시도한다.
- 입력 데이터를 임의로 바꿔 `Slime`에 hp를 추가한다.
- 실패 원인 분류 없이 코드를 반복 생성한다.

## AAVS-003: 모호한 요청에서 사용자 입력 재요청

### 목적

Agent가 불충분한 요청을 임의로 해석하지 않고, 필요한 추가 정보를 사용자에게 명확히 요청하는지 확인한다.

### 사용자 입력 1

```text
데이터 정리해줘.
```

### 기대 Agent 응답

```text
어떤 데이터인지, 어떤 방식으로 정리할지 알려주시겠어요? 예를 들어 중복 제거, 날짜 정렬, 누락값 처리 같은 기준이 필요합니다.
```

### 사용자 입력 2

```text
아래 CSV에서 중복 행을 제거하고 날짜 기준으로 오름차순 정렬해줘.
date,name,score
2026-04-03,Alice,10
2026-04-01,Bob,20
2026-04-03,Alice,10
2026-04-02,Charlie,15
```

### 기대 실행 흐름

1. 첫 번째 입력에서는 툴을 생성하지 않고 `clarification_requested` 이벤트를 남긴다.
2. 두 번째 입력을 받은 뒤 CSV 처리 툴을 생성한다.
3. 생성된 툴은 `csv` 표준 모듈 또는 구조화된 CSV 파서를 사용한다.
4. 중복 행을 제거한다.
5. `date` 기준으로 오름차순 정렬한다.
6. 결과를 사용자에게 표 형태 또는 CSV 형태로 제공한다.

### 기대 결과

```csv
date,name,score
2026-04-01,Bob,20
2026-04-02,Charlie,15
2026-04-03,Alice,10
```

### 수용 기준

- 모호한 첫 요청에서 임의 처리하지 않는다.
- 추가 질문은 사용자가 다음 행동을 알 수 있을 만큼 구체적이다.
- CSV 파싱은 문자열 split 또는 정규식이 아니라 표준 CSV 파서를 사용한다.
- 실제 생성 툴 실행 결과와 최종 응답이 일치한다.

### 실패 조건

- 첫 요청만으로 임의 데이터를 가정한다.
- 중복 제거 전에 정렬해 결과가 불안정해진다.
- 날짜를 문자열이 아닌 잘못된 형식으로 변환해 순서가 틀어진다.

## AAVS-004: 생성 툴 저장 동의 및 거부 처리

### 목적

Agent가 문제 해결 후 생성된 툴을 다음 세션에서도 재사용할 수 있도록 저장할지 사용자에게 확인하고, 사용자 선택에 따라 저장 또는 폐기하는지 확인한다.

### 사용자 입력

```text
아래 숫자 목록의 중앙값과 평균을 구해줘.
[10, 2, 7, 4, 9]
```

### 기대 실행 흐름

1. Agent가 숫자 목록 분석 툴을 생성한다.
2. 실제 실행으로 평균과 중앙값을 계산한다.
3. 최종 답변을 제공한다.
4. Agent가 생성 툴 저장 여부를 묻는다.

### 기대 최종 답변

```text
평균은 6.4이고 중앙값은 7입니다.
이 툴을 저장해두면 다음 세션에서도 사용할 수 있습니다. 저장할까요? (y/n)
```

### 분기 A: 사용자가 `y`를 입력

#### 수용 기준

- 툴이 실제 툴 라이브러리 저장 위치에 저장된다.
- 저장된 툴 메타데이터에는 최소한 이름, 설명, 입력 형식, 출력 형식, 생성 시각, 검증 결과가 포함된다.
- 같은 기능의 기존 툴이 있으면 중복 저장 대신 재사용 또는 병합 후보로 표시한다.
- 저장 완료 메시지를 제공한다.

### 분기 B: 사용자가 `n`을 입력

#### 수용 기준

- 생성된 임시 툴은 세션 종료 후 재사용 대상에 포함되지 않는다.
- 사용자가 거부했음을 기록한다.
- 저장하지 않았다는 메시지를 제공한다.

### 실패 조건

- 사용자 동의 없이 툴을 저장한다.
- 사용자가 `n`을 입력했는데도 라이브러리에 남긴다.
- 저장 여부 질문 없이 세션을 종료한다.
- 저장 메타데이터가 없어 재사용 가능성을 검증할 수 없다.

## AAVS-005: 저장된 툴 재사용과 중복 생성 방지

### 목적

한 번 저장된 툴이 다음 유사 요청에서 재사용되고, 같은 기능의 툴이 반복 생성되지 않는지 확인한다.

### 사전 조건

AAVS-004 분기 A를 먼저 통과해 숫자 통계 툴이 저장되어 있어야 한다.

### 사용자 입력

```text
아래 숫자들의 평균과 중앙값을 알려줘.
[3, 5, 11, 20, 21, 22]
```

### 기대 실행 흐름

1. Agent가 기존 툴 라이브러리를 조회한다.
2. 저장된 숫자 통계 툴을 후보로 선택한다.
3. 새 툴을 생성하지 않고 기존 툴을 실행한다.
4. 결과를 사용자에게 제공한다.

### 기대 결과

```text
평균은 13.6666666667이고 중앙값은 15.5입니다.
```

### 수용 기준

- 기존 툴 선택 근거가 기록된다.
- 동일 기능의 새 툴이 추가로 생성되지 않는다.
- 결과는 실제 실행 결과와 일치한다.

### 실패 조건

- 매번 새 툴을 생성해 라이브러리에 중복 저장한다.
- 저장된 툴의 입력 형식과 맞지 않는데도 강제로 실행한다.
- 재사용 실패 원인을 기록하지 않는다.

## AAVS-006: 자가 수정 중단과 사용자 개입 요청

### 목적

Agent가 해결 불가능하거나 정보가 부족한 실패를 무한히 자가 수정하지 않고, 적절한 시점에 사용자 개입을 요청하는지 확인한다.

### 사용자 입력

```text
내 비공개 데이터베이스에서 지난달 매출 상위 10개 상품을 분석해줘.
```

### 기대 실행 흐름

1. Agent가 데이터베이스 접속 정보가 없음을 인식한다.
2. 임의 접속 정보나 가짜 데이터를 만들지 않는다.
3. 필요한 정보를 사용자에게 요청한다.
4. 접속 정보가 없으면 툴 생성 또는 실행을 보류한다.

### 수용 기준

- Agent가 외부 리소스 접근 권한 부족을 명확히 설명한다.
- 필요한 정보 목록을 제시한다.
- mock 데이터로 결과를 꾸미지 않는다.
- 실패 분류는 `사용자 입력 부족` 또는 `실행 환경 오류`로 기록된다.

### 실패 조건

- 임의 매출 데이터를 생성해 분석 결과처럼 답한다.
- API 키, DB URL, 인증 정보를 추측한다.
- 접근 불가능한 상태에서 반복적으로 실행을 시도한다.

## 릴리즈 게이트

아래 조건을 만족해야 기본 동작 검증을 통과한 것으로 본다.
