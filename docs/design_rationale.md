# AdaptiveAgent 설계 근거

> 작성일: 2026-05-03  
> 목적: 이번 주 동안 고민한 설계 판단들을 사후에 정리한 기록. 코드 레벨 구현 방법보다 "왜 이 구조를 선택했는가"와 "어디에 한계가 있는가"를 솔직하게 남긴다.

---

## 1. 이게 뭘 만들려고 했나

한 줄로 요약하면 **"자연어 작업을 받아서 직접 코드를 짜고 실행하고, 잘 만든 도구는 기억해뒀다가 다음에 재사용하는 에이전트"** 다.

들어보면 단순한 것 같은데 실제로 만들다 보면 생각보다 결정할 것들이 많다. 계획은 LLM이 짜는데 실행은 Python subprocess로 한다. 실행이 실패하면 LLM이 다시 고친다. 근데 LLM이 고치는 것도 무한히 반복하면 안 된다. 사용자가 허락해야 저장이 된다. 저장된 건 다음 세션에서도 꺼낼 수 있어야 한다.

이 모든 게 맞물리는 구조를 어떻게 설계할지가 핵심이었다.

---

## 2. 설계 결정 사항

### 2-1. 외부 에이전트 프레임워크를 안 쓴 이유

처음 고민한 부분이다. LangChain이나 비슷한 프레임워크를 쓰면 빠르게 시작할 수 있는데, 결국 안 쓰기로 했다.

이유는 단순하다. **프레임워크가 제어 흐름을 가져가버리면 내가 원하는 걸 끼워 넣기 어려워진다.** HITL(Human-in-the-loop)처럼 실행 중에 멈추고 사용자 응답을 기다렸다가 재개하는 흐름, 또는 실행 결과를 보고 Critic이 재시도 vs 사용자 개입 요청을 결정하는 흐름은 프레임워크 위에서 구현하면 우회 코드가 쌓인다. 나중에 뭔가 고치려 할 때 프레임워크 내부 구조를 이해해야 하는 상황이 오는 게 더 피곤하다.

직접 만들면 느리지만 흐름 전체가 코드에 드러난다. 어디서 막히는지 추적하기도 쉽다. 이번 AAVS 검증 사이클에서 실패 원인을 "어느 노드에서 막혔나"로 정확히 잡을 수 있었던 것도 이 덕분이다.

---

### 2-2. StateMachineRouter + AgentState 조합

실행 흐름을 상태 기계로 표현한 이유는 **"지금 어디 있는가"를 항상 명확하게 만들기 위해서**다.

```
retrieve → plan → code → execute → critique
              ↑__________________________|  (retry)
              |__________________________|  (needs_human)
```

`AgentState`가 공유 상태를 들고 다니고, `StateMachineRouter`가 `next_node`를 보고 다음 역할을 결정한다. 각 노드는 상태를 읽고 쓰는 것만 한다. 서로 직접 호출하지 않는다.

이 구조의 장점은 **중간에 멈출 수 있다는 것**이다. HITL에서 사용자 응답이 올 때까지 상태를 저장하고 기다려야 하는데, 상태 기계면 스냅샷 찍고 나중에 거기서 재개하면 된다. 함수 호출 체인으로 만들었다면 이 재개가 훨씬 복잡해진다.

비판적으로 보면, 지금은 `AgentState`에 너무 많은 것들이 들어 있다. 공유 상태가 비대해지면 어느 노드가 언제 어떤 필드를 쓰는지 추적하기 어렵다. 나중에 노드가 늘어나거나 병렬 실행이 복잡해지면 상태 충돌 가능성이 있다.

---

### 2-3. Plan / Coder / Critic 역할 분리

처음에는 하나의 LLM 호출로 "계획도 짜고 코드도 써"를 시켰는데, 이게 맞지 않는다.

**계획(무엇을 할지)과 구현(어떻게 할지)은 다른 역량이다.** 특히 소형 로컬 모델에서는 컨텍스트가 길어질수록 포맷이 흔들린다. 계획은 짧은 JSON 하나면 충분한데, 코드까지 같이 내놓으라고 하면 JSON이 잘리거나 이중 인코딩이 나온다.

분리의 실제 효과:
- `Plan Agent`는 "어떤 도구를 쓸지"만 결정하면 된다. 짧은 JSON, 포맷 안정성 높음
- `Coder Agent`는 Python 코드만 집중해서 쓴다. 코딩 특화 모델로 교체하기도 쉬워진다
- `Critic Agent`는 실행 결과만 평가한다. retry / needs_human / failed 판정

이 분리가 있어야 **나중에 코더 역할에만 다른 모델을 꽂는 구조**가 가능해진다. 지금 qwen3.5 계열이 플래너는 잘 하는데 코더가 약한 게 실측으로 확인됐는데, 이 구조 덕분에 코더 역할만 `qwen2.5-coder`로 교체하는 게 비교적 단순한 작업이 된다.

---

### 2-4. 프롬프트를 코드 밖 파일로 관리

`plan.txt`, `coder.txt`, `critic.txt`, `correction.txt`를 모두 `prompts/default/` 아래 텍스트 파일로 뺐다.

이유는 **프롬프트 변경이 코드 배포 없이 가능해야 하기 때문**이다. 실제로 이번 AAVS 검증 사이클에서 프롬프트를 수십 번 고쳤는데, 매번 코드를 건드리지 않아도 됐다.

그리고 역할별 프롬프트가 독립 파일이면 **A/B 테스트가 쉽다**. `prompts/openai/coder.txt` vs `prompts/ollama-general/coder.txt` 같은 provider별 분기도 `PromptLoader`에 fallback 체인만 추가하면 된다.

단점은 프롬프트가 실제 어떻게 렌더링되는지 코드와 파일 두 군데를 봐야 한다는 것이다. 변수 이름이 바뀌면 파일과 코드가 불일치할 수 있다. 지금은 테스트가 이걸 잡아주고 있는데, 규모가 커지면 더 엄격한 관리가 필요해질 것이다.

---

### 2-5. tool_create → tool_validate → tool_approve 파이프라인

생성된 도구가 바로 사용되지 않고 반드시 이 파이프라인을 거치도록 설계했다.

```
tool_create   : 코드 파일만 생성. manifest 미등록
tool_validate : subprocess 샌드박스에서 AST 검사 + 샘플 실행 검증
tool_approve  : 사용자 승인 후 manifest.json 등록
```

**왜 이렇게 나눴냐면**, LLM이 생성한 코드를 신뢰해서 바로 실행시키면 안 된다고 판단했기 때문이다. 특히 자동으로 manifest에 등록해서 재사용하는 흐름에서는 잘못된 코드가 라이브러리에 쌓이는 문제가 생긴다. EvolveTool-Bench 논문에서도 이 점을 지적하고 있다 — 무분별하게 생성하면 중복과 버그가 쌓인다.

승인 단계를 사용자가 명시적으로 거치게 한 것은 **Human-in-the-loop의 가장 기본적인 구현**이다. "에이전트가 알아서 다 하는" 방향보다 "사용자가 결정권을 갖는" 방향을 우선했다.

비판적으로 보면, 지금 `tool_validate`가 샘플 인자 없이 호출되면 빈 dict로 `run({})`을 실행해서 실패한다는 버그가 있었다. 이번에 고치긴 했는데 (샘플 없으면 AST 검사만), 검증 강도를 낮춰야 했다는 점은 아직 아쉽다.

---

### 2-6. subprocess 샌드박스 — Docker를 의도적으로 빼놓은 이유

실행 환경 격리에는 subprocess `LocalSandboxBackend` 하나만 쓴다. Docker는 설계에서 일부러 제외했다.

이유는 **개발 환경이 Docker를 요구하지 않아야 한다고 판단했기 때문**이다. Docker가 있는 환경, 없는 환경, CI/CD, 로컬 개발, 서비스 배포 — 환경마다 다르다. 처음부터 Docker를 강제하면 환경 설정 비용이 올라가고, 빠른 피드백 사이클을 방해한다.

subprocess 샌드박스는 불완전하다. 생성 코드가 파일 시스템을 건드리거나 네트워크를 쓰는 걸 완전히 막지 못한다. 지금은 정책 기반 차단(`dangerous_shell_pattern`, `sensitive_absolute_path`)으로 어느 정도 방어하고 있지만, 이건 진짜 격리가 아니다.

하지만 **"지금 어떤 배포 환경에서 쓸 건지"가 결정되기 전에 특정 샌드박스 기술에 묶이는 건 좋지 않다**고 봤다. 나중에 환경이 확정되면 그때 Docker, gVisor, Firecracker 같은 강한 격리를 끼워 넣을 수 있도록 인터페이스만 정의해두는 방향을 택했다.

---

### 2-7. SkillCatalog — 장기 메모리의 핵심

`manifest.json`에 승인된 도구를 저장하고, 다음 세션에서 Top-K 검색으로 꺼내 쓰는 구조다.

이게 중요한 이유는 **에이전트가 "기억"을 갖게 되는 지점**이기 때문이다. 매번 새로 만드는 게 아니라, 이미 검증된 도구가 있으면 재사용한다. SkillX 논문이 말하는 방향이기도 하다.

지금은 키워드 + 태그 기반 Top-K 검색과 OpenAI embedding 코사인 유사도를 함께 쓴다. 도구가 적을 때는 이걸로 충분한데, 도구가 수백 개 쌓이면 검색 정확도 문제가 생길 수 있다.

`usage_count`와 `failure_count`를 기록하는 것도 여기서 하는데, 나중에 스킬이 오래되거나 품질이 나빠지면 자동으로 감지하기 위한 기반이다.

---

### 2-8. HITL — 멈추는 것도 설계다

생성 도구 승인, 위험한 작업 감지, 정보 부족 시 사용자 질문 — 이 세 가지 경우에 에이전트가 멈추고 사용자를 기다린다.

기술적으로는 실행 중 `pending` 상태를 `session_id`와 함께 저장하고, 나중에 `--resume`, `--approve`, `--reject`, `--input` 옵션으로 재개한다.

이걸 제대로 설계하는 게 생각보다 까다로웠다. 단순히 "질문하고 대답 받는" 게 아니라, **프로세스가 완전히 종료됐다가 다시 시작해도 이어지는 것**이 목표였다. 그래서 상태 스냅샷을 파일에 저장하고 복구하는 방식을 택했다.

지금은 단일 사용자 단일 세션만 지원한다. 사용자 구분, 권한, 다중 세션은 의도적으로 빼놓았다. "지금 당장 필요한 것"과 "나중에 필요한 것"을 구분하려고 했다.

## 3. 한계 및 개선 가능 방향

### 3-1. 긴 세션에서 컨텍스트 증가 문제

지금 구조에서 대화가 길어지면 `AgentState.reflections`, 이벤트 로그, `last_tool_result` 같은 것들이 계속 쌓인다. 이게 LLM 호출 시 컨텍스트로 들어가는데, 길어질수록 성능이 떨어지고 비용이 올라간다.

해결 방향은 두 가지다:

**Memory compact.** 오래된 reflection이나 이벤트 요약을 압축해서 짧게 만든다. 디테일은 잃지만 핵심 맥락은 유지한다. 지금 `SessionStore.cleanup()`이 TTL 기반으로 오래된 세션은 지우는데, 세션 내부 압축은 없다.

**Context budget 관리.** 모델별로 컨텍스트 한도가 다르다. qwen3.5 같은 로컬 모델은 길면 포맷이 무너진다는 게 AAVS 테스트에서 직접 확인됐다. 모델에 맞는 최대 컨텍스트를 알고, 그 범위 안에서 우선순위가 높은 정보만 넣는 컨트롤이 필요하다.

---

### 3-2. Self-evolve — 스킬이 스스로 개선되는 것

지금은 도구가 한 번 만들어지면 그대로 있다. 승인됐다고 끝이 아니다.

실제로 쓰다 보면 특정 입력에서는 잘 되는데 다른 입력에서는 실패하는 도구가 있다. 이걸 자동으로 감지하고 버전을 올리거나 대안 구현을 만들어두는 게 Self-evolve 개념이다.

`failure_count` 통계를 이미 기록하고 있는데, 이걸 활용하는 로직이 아직 없다. "failure가 n번 이상이면 도구 재작성을 제안"하는 정도의 트리거를 Librarian에 붙이는 게 첫 번째 단계가 될 것 같다.

AgentEvolver 논문이 말하는 "상향식 스킬 진화"는 훨씬 복잡한 이야기지만, 작게 시작하면 지금 구조 안에서도 가능하다.

---

### 3-3. Docker / 가상 환경 격리

앞서 말했듯이 일부러 빼놓은 부분이다. 그런데 빼놓은 게 한계라는 것도 인정해야 한다.

지금 subprocess 샌드박스는:
- 생성 코드가 파일 시스템 건드리는 걸 완전히 못 막는다
- 네트워크 접근도 정책으로만 막는다
- 서로 다른 Python 패키지 버전 충돌 문제가 있을 수 있다

**환경마다 답이 다르다는 게 핵심이다.** 로컬 개발자는 venv 하나면 충분하고, 프로덕션 서비스는 컨테이너 격리가 필수고, CI/CD는 또 다른 요구사항이 있다. 하나의 정답으로 통일하지 않고 인터페이스(`SandboxBackend` 추상화)만 정의해둔 건 맞는 방향이라고 생각한다.

나중에 환경이 정해지면 `DockerSandboxBackend`나 `gVisorSandboxBackend`를 꽂아 넣는 방식으로 확장하면 된다.

---

### 3-4. User Session — 지금은 단일 사용자

HITL 재개는 구현했는데, 사용자를 구분하는 개념이 없다. 지금은 모든 것이 하나의 workspace 기준이다.

여러 사용자가 쓰는 서비스 형태로 가려면:
- 사용자 구분 (auth)
- 사용자별 세션 격리
- 사용자별 스킬 라이브러리 (또는 공유 + 개인)
- 권한 관리

이건 "에이전트 기능"이 아니라 "서비스 인프라" 문제다. 지금 단계에서 이걸 설계 안에 넣으면 실제 에이전트 동작을 검증하기도 전에 인프라에 묻혀버린다고 판단해서 제외했다. 맞는 판단이었다고 생각한다.

---

### 3-5. 역할별 모델 분리 — Planner ≠ Coder

이번 AAVS 검증에서 나온 가장 실용적인 발견이다.

qwen3.5 계열을 전부 테스트했을 때, **플래너는 잘 하는데 코더가 약하다**는 게 데이터로 확인됐다. 7개 실패 전부가 Coder 단계에서 발생했다. 플래너가 `code_execute`나 `tool_create`를 올바르게 선택하는 비율은 높다.

지금 구조는 모든 역할이 동일한 모델을 쓴다. `config.py`에 `coder_model`을 분리하고 `_code_with_llm`에 별도 클라이언트를 주입하면 **"플래너는 작은 모델(빠르고 싸게), 코더는 코딩 특화 모델"** 조합이 가능해진다.

```
플래너: qwen3.5:2b (로컬, 빠름)
코더:   qwen2.5-coder:7b 또는 deepseek-coder-v2 (코딩 특화)
크리틱: qwen3.5:2b (로컬, 빠름)
```

이 방향이 성능 대비 비용 트레이드오프에서 가장 현실적인 다음 단계로 보인다.

---

### 3-6. Critic의 한계 — 실행 성공과 정답은 다르다

Critic이 지금 하는 것: "실행이 성공했고 JSON 출력이 나왔으면 success"

Critic이 못 하는 것: "출력값이 태스크 요구사항과 실제로 맞는지"

AAVS-007에서 qwen이 틀린 집계값(`total=190`, 실제 `350`)을 냈는데 Critic이 success로 통과시켰다. 코드가 실행됐고 JSON이 나왔으니까. 이건 Critic 설계의 근본적 한계다.

완전한 해결은 어렵다. Critic이 모든 태스크의 정답을 알 수는 없다. 하지만 "원본 task 텍스트와 실행 결과를 함께 주고 합리성을 확인하라"는 프롬프트 보강은 가능하다. 100%는 아니어도 명백한 오류(숫자 범위 벗어남, 빈 결과 등)는 잡을 수 있다.

## 마무리

이번 주 동안 만든 것 자체보다 **"실제로 테스트해보면서 어디가 왜 깨지는지"를 배운 것**이 더 가치 있었다. 

AAVS 16개 시나리오가 생각보다 날카롭게 시스템의 빈틈을 찔렀다. Critic이 retry를 잘못 내리는 것, JSON 파서가 LLM 출력의 이상한 패턴에 무너지는 것, 소형 모델이 프롬프트를 어떻게 오해하는 것 — 이런 것들이 코드를 짜기 전에는 보이지 않았다.

비판적으로 보면 아직 **"실제 사용 시나리오에서의 장기 동작"은 전혀 검증이 안 됐다**. 16개 시나리오는 단발 태스크다. 스킬 라이브러리가 수십 개 쌓이면 검색이 제대로 되는지, 세션이 길어지면 성능이 어떻게 변하는지, 동시에 여러 태스크를 처리하면 상태가 꼬이지 않는지 — 이건 다 열린 질문이다.

지금 만든 것은 **"핵심 루프가 동작하는 검증 가능한 기반"**이다. 다음은 이 기반 위에서 하나씩 검증하고 확장하는 단계다.
